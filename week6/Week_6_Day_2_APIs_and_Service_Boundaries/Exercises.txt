Answer in your own words:

1. What is an API? An Application Programming Interface is a set of rules and protocols that enables
 different software applications to communicate with each other and exchange data. 
 It acts as a messenger or intermediary, allowing one application to request services or information 
 from another without needing to know the technical details of how the other system is built internally. 

2. Why do we use APIs instead of directly accessing databases? A database API, such as PostgreSQL or Oracle is 
thoroughly tested and understood. There is no need to re-invent a database interface, when there are many good choices
already available, saving time and resources. Using a well-established database can also help with reliability and security, 
again because it is thoroughly tested and well-understood.

3. Give one real-world analogy for an API. In a more general sense, I have seen an interface control document, which can
describe an API, but can also describe other types of interfaces: electrical, RF, mechanical, anything. 
So, a major part of an API would be documentation that describes the API. Another part would be the public-facing software 
interfaces, which should be described by the documentation.

        - Database queries, possibly made user friendly with a website and/or GUI.

    - Results from queries
        - Previously entered user input: Company names, application status, contact information
- What data should never flow directly to the UI?
    - UI for non-technical users:
        - Hidden key(s) in the database
        - Errors that are difficult to understand. If an error is expected, such as invalid user input, 
            it should be explained so that a non-technical user can understand what went wrong and how to
            correctly format input. Another example of an expected error is the database being unavailable. 
            The user should be informed that the service is temporarily unavailable, try again later.
            Unexpected errors can be passed to the user, if the language has thorough error coverage, like rust.
            Other languagues, such as Python, do not have such protections, so an error that is difficult to understand or
            meaningless to a non-technical user should be logged and reported to a bug tracker.
    - UI for technical users - developers, admins - never say never! All information should be available. That's assuming that
        error logs and bug reports are part of the UI.

üß† Concept Check (Write as Comments)
Answer in 1‚Äì2 sentences each:
Why are APIs considered contracts? An API often is the interface between two independent organizations, such as companies. 
    In that case, a contract between the two groups will include the API or a description of interfaces.
What breaks when an API changes unexpectedly? Everyone who was not informed of the change is at risk of something breaking.
Why is backward compatibility important? That's related to the previous question about what breaks when an API changes 
    unexpectedly. If the change is backward-compatible, it won't break anything. For example, keeping all existing code the
    same, but adding features. If existing features are deprecated without warning, then whoever uses those features
    loses compatibility.
